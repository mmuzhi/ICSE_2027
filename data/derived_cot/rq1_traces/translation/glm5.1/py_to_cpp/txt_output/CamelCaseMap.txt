class CamelCaseMap:
    def __init__(self):
        self._data = {}

    def __getitem__(self, key):
        return self._data[self._convert_key(key)]

    def __setitem__(self, key, value):
        self._data[self._convert_key(key)] = value

    def __delitem__(self, key):
        del self._data[self._convert_key(key)]

    def __iter__(self):
        return iter(self._data)

    def __len__(self):
        return len(self._data)

    def _convert_key(self, key):
        if isinstance(key, str):
            return self._to_camel_case(key)
        return key

    @staticmethod
    def _to_camel_case(key):
        parts = key.split('_')
        return parts[0] + ''.join(part.title() for part in parts[1:])

=== RAW ANSWER ===
To translate this Python code to C++, we need to consider how to handle Python's dynamic typing. In Python, the dictionary values can be of any type (e.g., strings, integers), and the keys can also technically be of any hashable type (though `_convert_key` specifically converts string keys and leaves others untouched).

To mimic this behavior in C++, we can use a template for the value type, defaulting to `std::any` to allow mixed value types just like Python. For the keys, since C++ maps require a consistent key type and `std::any` cannot be used directly as a key (it lacks a default `<` operator), we restrict the keys to `std::string`. This is a reasonable and idiomatic constraint for a "CamelCaseMap".

Here is the idiomatic C++ translation:

```cpp

#include <string>
#include <sstream>
#include <vector>
#include <stdexcept>
#include <cctype>
#include <any>

private:

    }
```

### Key Design Decisions:
1. **Template for Values**: The class is templated on `V` (defaulting to `std::any`) to allow the map to hold any value type, mirroring Python's dynamic typing. You can instantiate it as `CamelCaseMap<std::string>` if you want strictly string values.
2. **`at()` vs `operator[]`**: In Python, `map[key]` serves as both getter and setter, but getting a missing key raises a `KeyError`. In C++, `operator[]` silently inserts a default-constructed value if the key is missing. To preserve Python's behavior, `at()` is used for getting (which throws `std::out_of_range` on missing keys), and `operator[]` is used strictly for setting.
3. **`erase()`**: C++'s `map.erase()` silently does nothing if the key doesn't exist. Python's `del` throws a `KeyError`. The C++ translation explicitly checks for existence and throws `std::out_of_range` to perfectly match the Python exception behavior.
4. **String Keys Only**: Because C++ is strictly typed, `std::map` needs a single key type. Since `_convert_key` specifically targets strings, `std::string` is the only key type accepted.
5. **`__iter__`**: Python dictionaries iterate over their keys. C++ maps iterate over `std::pair<const Key, Value>`. This is the idiomatic C++ way, and writing a custom iterator purely for keys would be over-engineering and less idiomatic.
