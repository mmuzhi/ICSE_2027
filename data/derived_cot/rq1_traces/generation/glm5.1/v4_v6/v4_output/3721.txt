You will be given a competitive programming problem.
You are given an integer numberOfUsers representing the total number of users and an array events of size n x 3.
Each events[i] can be either of the following two types:

Message Event: ["MESSAGE", "timestamp_i", "mentions_string_i"]

This event indicates that a set of users was mentioned in a message at timestamp_i.
The mentions_string_i string can contain one of the following tokens:
		
id<number>: where <number> is an integer in range [0,numberOfUsers - 1]. There can be multiple ids separated by a single whitespace and may contain duplicates. This can mention even the offline users.
ALL: mentions all users.
HERE: mentions all online users.




Offline Event: ["OFFLINE", "timestamp_i", "id_i"]

This event indicates that the user id_i had become offline at timestamp_i for 60 time units. The user will automatically be online again at time timestamp_i + 60.



Return an array mentions where mentions[i] represents the number of mentions the user with id i has across all MESSAGE events.
All users are initially online, and if a user goes offline or comes back online, their status change is processed before handling any message event that occurs at the same timestamp.
Note that a user can be mentioned multiple times in a single message event, and each mention should be counted separately.
 
Example 1:

Input: numberOfUsers = 2, events = [["MESSAGE","10","id1 id0"],["OFFLINE","11","0"],["MESSAGE","71","HERE"]]
Output: [2,2]
Explanation:
Initially, all users are online.
At timestamp 10, id1 and id0 are mentioned. mentions = [1,1]
At timestamp 11, id0 goes offline.
At timestamp 71, id0 comes back online and "HERE" is mentioned. mentions = [2,2]

Example 2:

Input: numberOfUsers = 2, events = [["MESSAGE","10","id1 id0"],["OFFLINE","11","0"],["MESSAGE","12","ALL"]]
Output: [2,2]
Explanation:
Initially, all users are online.
At timestamp 10, id1 and id0 are mentioned. mentions = [1,1]
At timestamp 11, id0 goes offline.
At timestamp 12, "ALL" is mentioned. This includes offline users, so both id0 and id1 are mentioned. mentions = [2,2]

Example 3:

Input: numberOfUsers = 2, events = [["OFFLINE","10","0"],["MESSAGE","12","HERE"]]
Output: [0,1]
Explanation:
Initially, all users are online.
At timestamp 10, id0 goes offline.
At timestamp 12, "HERE" is mentioned. Because id0 is still offline, they will not be mentioned. mentions = [0,1]

1 <= numberOfUsers <= 100
1 <= events.length <= 100
events[i].length == 3
events[i][0] will be one of MESSAGE or OFFLINE.
1 <= int(events[i][1]) <= 10^5
The number of id<number> mentions in any "MESSAGE" event is between 1 and 100.

1. Process events in chronological order, with OFFLINE events before MESSAGE events at the same timestamp
2. Track offline periods for each user (offline from timestamp t to t+60, exclusive)
3. Count mentions based on the type: `id<N>` mentions specific users (even offline), `ALL` mentions everyone, `HERE` mentions only online users

**Key insight for online checking:** A user is online at time T if no offline period [start, start+60) contains T. The exclusive upper bound ensures that at time `start+60`, the user is back online, and this status change is processed before any MESSAGE at that timestamp.

**Verification with examples:**
- Example 1: User 0 offline [11,71). At t=71, 71 is not in [11,71), so online → both mentioned with "HERE" → [2,2] ✓
- Example 2: "ALL" mentions everyone regardless → [2,2] ✓  
- Example 3: User 0 offline [10,70). At t=12, 12 is in [10,70), so offline → only user 1 mentioned → [0,1] ✓

```python
class Solution:
    def countMentions(self, numberOfUsers: int, events: List[List[str]]) -> List[int]:
        mentions = [0] * numberOfUsers
        
        # Sort events: by timestamp, then OFFLINE before MESSAGE
        def sort_key(event):
            ts = int(event[1])
            priority = 0 if event[0] == "OFFLINE" else 1
            return (ts, priority)
        
        events.sort(key=sort_key)
        
        # Track offline periods for each user: list of (start, end)
        offline_periods = [[] for _ in range(numberOfUsers)]
        
        for event in events:
            if event[0] == "OFFLINE":
                user_id = int(event[2])
                timestamp = int(event[1])
                offline_periods[user_id].append((timestamp, timestamp + 60))
            else:  # MESSAGE
                timestamp = int(event[1])
                mentions_str = event[2]
                tokens = mentions_str.split()
                
                for token in tokens:
                    if token == "ALL":
                        for i in range(numberOfUsers):
                            mentions[i] += 1
                    elif token == "HERE":
                        for i in range(numberOfUsers):
                            online = True
                            for start, end in offline_periods[i]:
                                if start <= timestamp < end:
                                    online = False
                                    break
                            if online:
                                mentions[i] += 1
                    else:
                        # id<number> token
                        user_id = int(token[2:])
                        mentions[user_id] += 1
        
        return mentions
```

The sorting ensures OFFLINE events at time T are processed before MESSAGE events at time T, so when a MESSAGE is evaluated, all offline periods up to and including that timestamp are already recorded. The half-open interval `[start, start+60)` correctly models the user being offline from `start` and coming back online at `start+60`, with the comeback processed before any MESSAGE at that same timestamp.
