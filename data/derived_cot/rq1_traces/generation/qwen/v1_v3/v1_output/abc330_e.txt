To solve this problem, we need to efficiently compute the mex (minimum excluded value) of an array after each update operation. The mex is the smallest non-negative integer not present in the array. Given the constraints, a brute-force approach would be too slow, so we use a segment tree to maintain the frequency of numbers in the range [0, N] (where N is the length of the array) and query the smallest missing number efficiently.

### Approach
1. **Problem Analysis**: The mex of an array is the smallest non-negative integer not present in the array. The challenge is to compute this efficiently after each update. The array can be large (up to 200,000 elements), and each query requires updating an element and then finding the mex.
2. **Intuition**: The mex must be in the range [0, N] because the array has N elements, and the set {0, 1, ..., N} contains N+1 distinct numbers. Thus, at least one number in this range is missing.
3. **Algorithm Selection**: We use a segment tree to keep track of the frequency of each number in the range [0, N]. The segment tree helps in querying the smallest number with zero frequency (i.e., the mex) in logarithmic time.
4. **Optimization**: By focusing only on numbers in [0, N], we ignore numbers outside this range, which do not affect the mex. The segment tree is built to store the minimum frequency in each segment, allowing efficient updates and queries.
5. **Complexity Analysis**: Each update and query operation on the segment tree takes O(log N) time. With Q queries, the total time complexity is O(Q log N), which is efficient for the given constraints.
