You will be given a competitive programming problem.
There are N mochi (rice cakes), arranged in ascending order of size.
The size of the i-th mochi (1\leq i\leq N) is A_i.
Given two mochi A and B, with sizes a and b respectively, you can make one kagamimochi (a stacked rice cake) by placing mochi A on top of mochi B if and only if a is at most half of b.
You are given Q integer pairs. Let (L_i, R_i) be the i-th pair (1\leq i\leq Q), and solve the following problem for each i:

Using only the R_i - L_i + 1 mochi from the L_i-th to the R_i-th, how many kagamimochi can you make simultaneously?  
More precisely, find the maximum non-negative integer K such that:

- Out of the R_i - L_i + 1 mochi from the L_i-th to the R_i-th, choose 2K mochi and form K pairs. For each pair, place one mochi on top of the other, to make K kagamimochi.

Input

The input is given from Standard Input in the following format:
N
A_1 A_2 \dotsc A_N
Q
L_1 R_1
L_2 R_2
\vdots
L_Q R_Q

Output

Print Q lines. The i-th line (1\leq i\leq Q) should contain the answer to the i-th query.

Constraints


- 2 \leq N \leq 2 \times 10^5
- 1 \leq A_i \leq 10^9 \ (1 \leq i \leq N)
- A_i \leq A_{i+1} \ (1 \leq i < N)
- 1 \leq Q \leq 2 \times 10^5
- 1 \leq L_i < R_i \leq N \ (1 \leq i \leq Q)
- All input values are integers.

To solve this problem, we need to determine the maximum number of kagamimochi (stacked rice cakes) that can be formed from a segment of sorted mochi (rice cakes) for each query. The key condition for forming a kagamimochi is that the size of the top mochi must be at most half the size of the bottom mochi. 

### Approach
1. **Problem Analysis**: Given a sorted array of mochi sizes, for each query (L, R), we need to find the maximum number of pairs (top, bottom) such that the top mochi is from the first half of the segment (L to mid) and the bottom mochi is from the second half (mid to R), and the size of the top mochi is at most half the size of the bottom mochi.
2. **Intuition**: Since the array is sorted, we can use a greedy approach. For each segment defined by the query, we iterate through the bottom mochis (from mid to R) and try to match them with the top mochis (from L to mid) in increasing order. The goal is to maximize the number of valid pairs.
3. **Algorithm Selection**: For each query, we calculate the starting index of the bottom mochis (mid). We then iterate through the bottom mochis, and for each, if the current top mochi (starting from the smallest) satisfies the condition (top ≤ bottom / 2), we pair them and move to the next top mochi. We break early if we have exhausted all possible pairs (i.e., when the number of pairs equals the number of bottom mochis).
4. **Complexity Analysis**: The worst-case time complexity per query is O(m), where m is the length of the segment (R - L + 1). For Q queries, the total time complexity is O(Q * m). In the worst case, this could be O(N * Q), which is manageable given the constraints (N, Q ≤ 2 × 10^5) with optimizations like early termination.

### Solution Code
```python
import sys

def main():
    data = sys.stdin.read().split()
    n = int(data[0])
    A = list(map(int, data[1:1+n]))
    q = int(data[1+n])
    queries = []
    index = 1 + n + 1
    for _ in range(q):
        L = int(data[index])
        R = int(data[index+1])
        index += 2
        queries.append((L-1, R-1))
    
    out_lines = []
    for (l, r) in queries:
        m = r - l + 1
        bases_start = l + (m + 1) // 2
        tops_count = (m + 1) // 2
        bases_count = m // 2
        k = 0
        for j in range(bases_start, r + 1):
            if k < tops_count and A[l + k] * 2 <= A[j]:
                k += 1
            if k == bases_count:
                break
        out_lines.append(str(k))
    
    sys.stdout.write("\n".join(out_lines))

if __name__ == '__main__':
    main()
```

2. **Processing Queries**: For each query, the segment (L, R) is converted to 0-based indices. The segment length (m) is calculated, and the starting index of the bottom mochis (bases_start) is determined as the midpoint of the segment.
3. **Greedy Matching**: We iterate through each bottom mochi in the segment. For each bottom mochi, we check if the current top mochi (starting from the smallest in the top half) satisfies the condition (top ≤ bottom / 2). 4. **Output**: The results for all queries are collected and printed at once for efficiency.
