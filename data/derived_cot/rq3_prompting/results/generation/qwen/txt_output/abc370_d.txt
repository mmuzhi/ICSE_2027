To solve this problem, we need to efficiently handle a series of queries on a grid where each query involves removing walls either at a specific cell or in the four cardinal directions from that cell. The challenge is to process these queries in a way that optimizes performance, given the constraints that the grid can be up to 400,000 cells and there can be up to 200,000 queries.

### Approach
1. **Initialization**: We start by reading the grid dimensions (H rows and W columns) and the number of queries (Q). We then initialize two data structures:
   - `row_list`: A list where each entry corresponds to a row and contains a sorted list of columns that still have walls in that row.
   - `col_list`: A list where each entry corresponds to a column and contains a sorted list of rows that still have walls in that column.

2. **Query Processing**: For each query, we check if the specified cell (R_q, C_q) has a wall. If it does, we remove it from both `row_list` and `col_list` and decrement the total wall count. If the cell does not have a wall, we look in the four directions (up, down, left, right) to find the nearest walls and remove them if they exist. The search for nearest walls is optimized using binary search (via `bisect`) on the sorted lists in `row_list` and `col_list`.

3. **Efficiency**: The use of sorted lists and binary search ensures that each query is processed in logarithmic time relative to the grid dimensions. However, the removal operation from a list is O(n) in the worst case. Given the constraints, this approach is feasible for smaller grids but may not be optimal for the upper limits. For larger grids, a more sophisticated data structure like a segment tree or Fenwick tree would be necessary, but this solution prioritizes clarity and correctness for the given problem constraints.
