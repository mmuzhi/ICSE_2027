There is an undirected graph with N vertices and 0 edges. The vertices are numbered 1 to N.
You are given Q queries to process in order. Each query is of one of the following two types:

- Type 1: Given in the format 1 u v. Add an edge between vertices u and v.
- Type 2: Given in the format 2 v k. Print the k-th largest vertex number among the vertices connected to vertex v. If there are fewer than k vertices connected to v, print -1.

The i-th line should contain the answer to the i-th Type 2 query.

- Problem Analysis:
  - Undirected graph, N vertices, 0 edges initially.
  - Q queries.
  - Type 1: Add edge between u and v. (u < v)
  - Type 2: Print k-th largest vertex connected to v. If fewer than k, print -1.
  - Constraints: N, Q <= 2e5, k <= 10.
  - Note: k is small (<= 10). This is a huge hint.

- Data Structure Candidate:
  - DSU (Disjoint Set Union) is natural for connectivity.
  - We need to find the k-th largest vertex in a component. Since k <= 10, we can maintain the top 10 largest vertices for each DSU root.
  - When merging two components, we can just merge their top 10 lists, sort, and keep the top 10.
  - This perfectly fits the constraints! k <= 10 makes merging top-10 lists O(k) = O(10) per union, which is extremely fast.

- Feasibility Screen:
  - Complexity: Q unions * 10 log 10 = Q * ~30 operations. Total time < 1e7 ops, well within 2 sec limit in Python.
  - Space: N * 10 integers for the lists. N <= 2e5, so 2e6 integers, well within memory limits.
  - Edge cases: Finding k-th largest when component size < k. Handled by checking `len(lst) >= k`.
  - DSU path compression and union by size/rank are standard. We need to store the top-10 list at the root.

- Implementation Ledger:
  - `parent` array for DSU.
  - `size` array for DSU (optional, but good for union by size).
  - `top_k` array of lists. `top_k[i]` stores the top 10 largest vertices in the component rooted at `i`.
  - `find(u)`: standard path compression.
  - `union(u, v)`: find roots `ru` and `rv`. If same, return. Attach smaller tree to larger tree. Merge `top_k[ru]` and `top_k[rv]`, sort descending, take first 10, assign to the new root.
  - Query Type 2: `v, k`. Find root `rv = find(v)`. If `len(top_k[rv]) >= k`, answer is `top_k[rv][k-1]`. Else `-1`.

- Refining the top_k list:
  - Initially, `top_k[i] = [i]`.
  - When merging, `merged_list = top_k[ru] + top_k[rv]`. Sort descending. `top_k[new_root] = merged_list[:10]`.
  - Wait, `top_k` elements are vertex numbers. Since vertices are 1 to N, larger vertex numbers mean larger values. So sorting descending and taking top 10 is correct.
  - Is it always true that the top 10 of the merged set is in the top 10 of the two subsets? Yes, obviously. The overall top 10 must be formed by the top elements of the two subsets.

- Local risks:
  - DSU path compression might mess up `top_k` if we aren't careful. `top_k` should ONLY be read/written at the root. During `find`, we update `parent`, but we shouldn't touch `top_k` for non-root nodes (or we just leave their old lists to be garbage collected, which is fine).
  - When doing union by size, we make one root the parent of another. We must update `top_k` for the NEW root, and we don't need to clear `top_k` for the old root (it just becomes inaccessible or ignored).

  def solve():
      input = sys.stdin.read
