Problem: Max number of queries to remove such that remaining queries can make `nums` a zero array. Each query `[l, r]` can decrement any index in `[l, r]` by at most 1.
This is equivalent to: we need to select a subset of queries to keep, such that for each index `i`, the number of selected queries covering `i` is at least `nums[i]`. We want to maximize the number of queries *removed*, which means minimizing the number of queries *kept*.
Wait, is it just minimizing the number of kept queries? Yes.
So we want to find the minimum size subset of queries such that for each `i`, the number of selected queries covering `i` is at least `nums[i]`.
This is a classic minimum set cover / hitting set variant, but with intervals and covering requirements.
Since we want to minimize the number of selected intervals, we should process indices from left to right, and greedily select intervals that end as far right as possible to cover the current index's requirement.
Let's verify this greedy approach.
For index `i`, we need at least `nums[i]` intervals covering `i`. We have some intervals that start at or before `i` and end at or after `i`. We should pick the ones that end the furthest to the right, because they can also cover future indices, potentially reducing the need to pick more intervals later.
Wait, if we just pick intervals greedily by their right endpoint, is it optimal?

Wait, the greedy strategy is:
Sort queries by left endpoint.
Use a max-heap for right endpoints of available queries.
Iterate `i` from 0 to `n-1`:
  Add all queries starting at `i` to the max-heap.
  While current coverage at `i` < `nums[i]`:
    Pop the query with the largest right endpoint from the heap.
    If its right endpoint < `i`, it's useless, return -1.
    Increment coverage at `i`.
    But wait! If we just increment coverage, how do we track coverage for future indices?
    We can use a difference array to add 1 to the range `[L, R]` of the selected query.
    Actually, we only need to know the active coverage at `i`.
    Let `diff` array be of size `n+1`.
    `cur_coverage += diff[i]`
    When we select a query `[L, R]`, we do `diff[L] += 1`, `diff[R+1] -= 1`.
    Since we process `L = i` currently, `diff[L] += 1` will increase `cur_coverage` immediately.
    And we need to add `diff[R+1] -= 1` to decrease coverage when we pass `R`.

Let's refine the algorithm:
1. Sort `queries` by left endpoint.
2. `diff = [0] * (n + 1)`
3. `cur_coverage = 0`
4. `pq = []` (max-heap of right endpoints)
5. `q_idx = 0`
6. `selected_count = 0`
7. For `i` from 0 to `n-1`:
   a. `cur_coverage += diff[i]`
   b. While `q_idx < len(queries)` and `queries[q_idx][0] == i`:
        `heapq.heappush(pq, -queries[q_idx][1])`
        `q_idx += 1`

        If `R < i`: return -1

        `diff[R + 1] -= 1`

Wait, is it always optimal to pick the query with the largest right endpoint?
Suppose we need to cover index `i`, and we have two available queries: one ending at `R1` and one ending at `R2` with `R1 > R2`.
If we pick the one ending at `R2`, we might run out of queries to cover `R1` later, or we might just need to pick another one for `R1` anyway. Picking the one ending at `R1` covers everything `R2` covers and potentially more. It strictly dominates `R2` in terms of future coverage. So picking the one with the largest `R` is always at least as good as picking a smaller `R`. This is a standard greedy proof for minimum interval covering.

Let's review the code structure.
```python
class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        diff = [0] * (n + 1)
        pq = []
        q_idx = 0
        selected_count = 0
        cur_coverage = 0
        
        for i in range(n):
            cur_coverage += diff[i]
            while q_idx < len(queries) and queries[q_idx][0] == i:
                heapq.heappush(pq, -queries[q_idx][1])
                q_idx += 1
                
            while cur_coverage < nums[i]:
                while pq and -pq[0] < i:
                    heapq.heappop(pq)
                if not pq:
                    return -1
                R = -heapq.heappop(pq)
                cur_coverage += 1
                diff[R + 1] -= 1
                selected_count += 1
                
        return len(queries) - selected_count
```

Is there any case where we need to pop from `pq` but it's empty, so we return -1, but there were queries that start later?
If `pq` is empty, it means there are no available queries that start at or before `i` and end at or after `i`.
Any query that starts later cannot cover index `i`.
Since we need to cover index `i` right now, and no available query can cover it, it's impossible to form a zero array. Returning -1 is correct.

Let's do a sanity check on the implementation details.
```python
import heapq

class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        diff = [0] * (n + 1)
        pq = []
        q_idx = 0
        selected_count = 0
        cur_coverage = 0
        
        for i in range(n):
            cur_coverage += diff[i]
            while q_idx < len(queries) and queries[q_idx][0] == i:
                heapq.heappush(pq, -queries[q_idx][1])
                q_idx += 1
                
            while cur_coverage < nums[i]:
                while pq and -pq[0] < i:
                    heapq.heappop(pq)
                if not pq:
                    return -1
                R = -heapq.heappop(pq)
                cur_coverage += 1
                diff[R + 1] -= 1
                selected_count += 1
                
        return len(queries) - selected_count
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `pq` yet?
We only add queries to `pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `pq` when we processed `l` (or just now if `l == i`). Thus, `pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `cur_coverage += diff[i]`: `O(1)`
  `while q_idx < len(queries) and queries[q_idx][0] == i`:
    Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped at most once. Total `O(Q log Q)` over all `i`.
    The inner `while pq and -pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(n + Q)`, which is about `2 \times 10^5` integers, taking a few megabytes. Very safe.

```python
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
```

```python
import heapq

class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])

                heapq.heappush(available_pq, -queries[q_idx][1])
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `available_pq` yet?
We only add queries to `available_pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `available_pq` when we processed `l` (or just now if `l == i`). Thus, `available_pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `while active_pq and active_pq[0] < i`: Each selected query is popped at most once. Total `O(Q log Q)` over all `i`.
  `while q_idx < len(queries) and queries[q_idx][0] == i`: Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped from `available_pq` at most once. Total `O(Q log Q)` over all `i`.
    The inner `while available_pq and -available_pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(Q)`, which is about `10^5` integers, taking a few megabytes. Very safe.

Is there any case where we need to pop from `available_pq` but it's empty, so we return -1, but there were queries that start later?
If `available_pq` is empty, it means there are no available queries that start at or before `i` and end at or after `i`.

Since we need to cover index `i` right now, and no available query can cover it, it's impossible to form a zero array. Returning -1 is correct.

Let's do a sanity check on the implementation details.
```python
import heapq

class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        available_pq = [] # max-heap of right endpoints
        active_pq = []    # min-heap of right endpoints of selected queries
        q_idx = 0
        selected_count = 0
        
        for i in range(n):
            while active_pq and active_pq[0] < i:
                heapq.heappop(active_pq)
            
            cur_coverage = len(active_pq)
            
            while q_idx < len(queries) and queries[q_idx][0] == i:
                heapq.heappush(available_pq, -queries[q_idx][1])
                q_idx += 1
                
            while cur_coverage < nums[i]:
                while available_pq and -available_pq[0] < i:
                    heapq.heappop(available_pq)
                if not available_pq:
                    return -1
                R = -heapq.heappop(available_pq)
                heapq.heappush(active_pq, R)
                cur_coverage += 1
                selected_count += 1
                
        return len(queries) - selected_count
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `available_pq` yet?
We only add queries to `available_pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `available_pq` when we processed `l` (or just now if `l == i`). Thus, `available_pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `while active_pq and active_pq[0] < i`: Each selected query is popped at most once. Total `O(Q log Q)` over all `i`.
  `while q_idx < len(queries) and queries[q_idx][0] == i`: Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped from `available_pq` at most once. Total `O(Q log Q)` over all `i`.
    The inner `while available_pq and -available_pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(Q)`, which is about `10^5` integers, taking a few megabytes. Very safe.

```python
class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        available_pq = [] # max-heap of right endpoints
        active_pq = []    # min-heap of right endpoints of selected queries
        q_idx = 0
        selected_count = 0
```

```python
import heapq

class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)

        active_pq = []    # min-heap of right endpoints of selected queries

                heapq.heappop(active_pq)

                heapq.heappush(available_pq, -queries[q_idx][1])
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `available_pq` yet?
We only add queries to `available_pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `available_pq` when we processed `l` (or just now if `l == i`). Thus, `available_pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `while active_pq and active_pq[0] < i`: Each selected query is popped at most once. Total `O(Q log Q)` over all `i`.
  `while q_idx < len(queries) and queries[q_idx][0] == i`: Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped from `available_pq` at most once. Total `O(Q log Q)` over all `i`.
    The inner `while available_pq and -available_pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(Q)`, which is about `10^5` integers, taking a few megabytes. Very safe.

Is there any case where we need to pop from `available_pq` but it's empty, so we return -1, but there were queries that start later?
If `available_pq` is empty, it means there are no available queries that start at or before `i` and end at or after `i`.
Any query that starts later cannot cover index `i`.
Since we need to cover index `i` right now, and no available query can cover it, it's impossible to form a zero array. Returning -1 is correct.

Let's do a sanity check on the implementation details.
```python
import heapq

class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        available_pq = [] # max-heap of right endpoints
        active_pq = []    # min-heap of right endpoints of selected queries
        q_idx = 0
        selected_count = 0
        
        for i in range(n):
            while active_pq and active_pq[0] < i:
                heapq.heappop(active_pq)
            
            cur_coverage = len(active_pq)
            
            while q_idx < len(queries) and queries[q_idx][0] == i:
                heapq.heappush(available_pq, -queries[q_idx][1])
                q_idx += 1
                
            while cur_coverage < nums[i]:
                while available_pq and -available_pq[0] < i:
                    heapq.heappop(available_pq)
                if not available_pq:
                    return -1
                R = -heapq.heappop(available_pq)
                heapq.heappush(active_pq, R)
                cur_coverage += 1
                selected_count += 1
                
        return len(queries) - selected_count
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `available_pq` yet?
We only add queries to `available_pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `available_pq` when we processed `l` (or just now if `l == i`). Thus, `available_pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `while active_pq and active_pq[0] < i`: Each selected query is popped at most once. Total `O(Q log Q)` over all `i`.
  `while q_idx < len(queries) and queries[q_idx][0] == i`: Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped from `available_pq` at most once. Total `O(Q log Q)` over all `i`.
    The inner `while available_pq and -available_pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(Q)`, which is about `10^5` integers, taking a few megabytes. Very safe.

```python
class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        available_pq = [] # max-heap of right endpoints
        active_pq = []    # min-heap of right endpoints of selected queries
        q_idx = 0
        selected_count = 0
```

```python
import heapq

class Solution:
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `available_pq` yet?
We only add queries to `available_pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `available_pq` when we processed `l` (or just now if `l == i`). Thus, `available_pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `while active_pq and active_pq[0] < i`: Each selected query is popped at most once. Total `O(Q log Q)` over all `i`.
  `while q_idx < len(queries) and queries[q_idx][0] == i`: Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped from `available_pq` at most once. Total `O(Q log Q)` over all `i`.
    The inner `while available_pq and -available_pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(Q)`, which is about `10^5` integers, taking a few megabytes. Very safe.

    pop -1 -> R=1. Push 1 to `active_pq`. `cur_coverage` = 1. `selected_count` = 2.
    pop -1 -> R=1. Push 1 to `active_pq`. `cur_coverage` = 2. `selected_count` = 3.
Result: 3 - 3 = 0. Correct.

What if `nums = [2, 1]`, `queries = [[0,1], [0,1], [1,1]]`
i=0:
  push -1, -1 to `available_pq`. `available_pq = [-1, -1]`
  `cur_coverage` = 0 < 2:
    pop -1 -> R=1. Push 1 to `active_pq`. `cur_coverage` = 1. `selected_count` = 1.
    pop -1 -> R=1. Push 1 to `active_pq`. `cur_coverage` = 2. `selected_count` = 2.
i=1:
  `active_pq` has 1, 1. None are `< 1`.
  `cur_coverage` = 2.
  push -1 to `available_pq`. `available_pq = [-1]`
  `cur_coverage` (2) < 1 -> False.
Result: 3 - 2 = 1. Correct.

What if `nums = [2, 1]`, `queries = [[0,0], [0,0], [1,1]]`
i=0:
  push -0, -0 to `available_pq`. `available_pq = [0, 0]`
  `cur_coverage` = 0 < 2:
    pop 0 -> R=0. Push 0 to `active_pq`. `cur_coverage` = 1. `selected_count` = 1.
    pop 0 -> R=0. Push 0 to `active_pq`. `cur_coverage` = 2. `selected_count` = 2.
i=1:
  `active_pq` has 0, 0. Both are `< 1`, so pop both.
  `cur_coverage` = 0.
  push -1 to `available_pq`. `available_pq = [-1]`
  `cur_coverage` < 1:
    pop -1 -> R=1. Push 1 to `active_pq`. `cur_coverage` = 1. `selected_count` = 3.
Result: 3 - 3 = 0. Correct.

Is there any case where we need to pop from `available_pq` but it's empty, so we return -1, but there were queries that start later?
If `available_pq` is empty, it means there are no available queries that start at or before `i` and end at or after `i`.
Any query that starts later cannot cover index `i`.
Since we need to cover index `i` right now, and no available query can cover it, it's impossible to form a zero array. Returning -1 is correct.

Let's do a sanity check on the implementation details.
```python
import heapq

class Solution:
    def maxRemoval(self, nums: List[int], queries: List[List[int]]) -> int:
        queries.sort(key=lambda x: x[0])
        n = len(nums)
        available_pq = [] # max-heap of right endpoints
        active_pq = []    # min-heap of right endpoints of selected queries
        q_idx = 0
        selected_count = 0
        
        for i in range(n):
            while active_pq and active_pq[0] < i:
                heapq.heappop(active_pq)
            
            cur_coverage = len(active_pq)
            
            while q_idx < len(queries) and queries[q_idx][0] == i:
                heapq.heappush(available_pq, -queries[q_idx][1])
                q_idx += 1
                
            while cur_coverage < nums[i]:
                while available_pq and -available_pq[0] < i:
                    heapq.heappop(available_pq)
                if not available_pq:
                    return -1
                R = -heapq.heappop(available_pq)
                heapq.heappush(active_pq, R)
                cur_coverage += 1
                selected_count += 1
                
        return len(queries) - selected_count
```

Wait, what if `nums[i]` is very large, say `10^5`, and we have enough queries, but they are not all in `available_pq` yet?
We only add queries to `available_pq` when their left endpoint is exactly `i`.
Is it possible that we need a query that starts at `i+1` to cover `i`?
No, because a query `[l, r]` covers indices from `l` to `r`. It cannot cover `i` if `l > i`.
So we only ever need queries with `l <= i` to cover `i`. All such queries are added to `available_pq` when we processed `l` (or just now if `l == i`). Thus, `available_pq` contains exactly the set of unused queries that can cover `i`.
This is perfectly correct.

Let's check the time complexity again.
`queries.sort()`: `O(Q log Q)`
Loop `i` from `0` to `n-1`:
  `while active_pq and active_pq[0] < i`: Each selected query is popped at most once. Total `O(Q log Q)` over all `i`.
  `while q_idx < len(queries) and queries[q_idx][0] == i`: Each query is pushed exactly once. Total `O(Q log Q)` over all `i`.
  `while cur_coverage < nums[i]`:
    Each query is popped from `available_pq` at most once. Total `O(Q log Q)` over all `i`.
    The inner `while available_pq and -available_pq[0] < i` also pops at most once per query.
So overall time complexity is `O(n + Q log Q)`.
With `n, Q <= 10^5`, `Q log Q` is about `1.6 \times 10^6` operations, which will run in less than 0.1 seconds in Python.
Space complexity is `O(Q)`, which is about `10^5` integers, taking a few megabytes. Very safe.

Let's consider if there's an alternative way to do this without `diff` array.
We could maintain a variable `active_selected` and when we select a query, we add it to a min-heap of right endpoints.
Then at the beginning of the loop for `i`, we pop from the min-heap all right endpoints that are `< i` and decrement `cur_coverage`.
