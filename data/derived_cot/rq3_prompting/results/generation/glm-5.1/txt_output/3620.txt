Constraints: n up to 1e5, nums[i] up to 1e9, k up to 1e9.

Candidate: Greedy. Sort the array. Try to assign each element to the smallest available distinct value.
Since we can add [-k, k] to `x`, the range of possible values for `x` is `[x - k, x + k]`.
If we process the sorted array, for each `x`, we want to choose the smallest integer `y` in `[x - k, x + k]` such that `y` is greater than the previously chosen value.
Let `prev` be the last chosen value.
For the current `x`, the minimum possible value we can choose is `max(x - k, prev + 1)`.
If this minimum possible value is `<= x + k`, we can choose it, increment the count of distinct elements, and set `prev = max(x - k, prev + 1)`.
If it's `> x + k`, we cannot choose a new distinct value for this `x`, so we skip it.

Wait, is it always optimal to choose the smallest available?
Yes, by standard interval scheduling / greedy proof. Choosing the smallest available leaves the maximum room for future elements.

```python
class Solution:
    def maxDistinctElements(self, nums: List[int], k: int) -> int:
        nums.sort()
        prev = -10**18
        count = 0
        for x in nums:
            min_val = max(x - k, prev + 1)
            if min_val <= x + k:
                count += 1
                prev = min_val
        return count
```
