# Chain-of-Thought Semantic Segmentation for Software Engineering Tasks

You are an expert in analyzing reasoning processes in software engineering tasks. Your task is to segment a Chain-of-Thought (CoT) into distinct reasoning steps and classify each step into high-level reasoning categories.

## Task
Given a CoT reasoning trace, you need to:
1. **Segment** the CoT into meaningful reasoning steps based on semantic boundaries
2. **Classify** each step using one of the 8 major categories below

---

## Reasoning Category Taxonomy

### 1. PROBLEM UNDERSTANDING (PU)
Understanding and clarifying the problem requirements.

**This category includes:**
- Parsing and interpreting problem statements
- Clarifying ambiguous or implicit requirements
- Analyzing given examples or test cases to understand the problem
- Identifying constraints, edge cases, or special conditions
- Extracting key information from problem description

**Example indicators:** "The problem asks...", "Given that...", "We need to...", "The constraint is...", "From the example..."

---

### 2. SOLUTION DESIGN (SD)
Designing the high-level solution approach and strategy.

**This category includes:**
- Proposing or exploring solution strategies
- Selecting algorithms (e.g., greedy, DP, brute force, divide-and-conquer)
- Choosing or designing data structures
- Breaking down complex problems into sub-problems
- Analyzing time/space complexity
- Exploring and comparing different approaches (even tentatively)

**Example indicators:** "The approach is...", "We can use...", "This is a ... problem", "Let's try...", "The algorithm should...", "The complexity is..."

---

### 3. IMPLEMENTATION PLANNING (IP)
Planning the detailed implementation steps and logic.

**This category includes:**
- Step-by-step implementation workflow planning
- Detailed implementation logic or pseudocode
- Planning how to handle edge cases in implementation
- Optimization considerations before coding
- Mapping high-level design to concrete implementation steps

**Example indicators:** "First, we...", "The steps are...", "We need to handle...", "The implementation should...", "To optimize..."

---

### 4. CODE CONSTRUCTION (CC)
Writing and refining the actual code.

**This category includes:**
- Writing code skeleton or structure
- Actual code implementation
- Refining or improving existing code
- Adding code comments or explanations

**Example indicators:** "The code is...", "def function()...", "```python", "Here's the implementation...", "Let me write..."

---

### 5. VERIFICATION & VALIDATION (VV)
Testing, debugging, and validating the solution.

**This category includes:**
- Manual tracing through algorithm with examples (walkthrough)
- Testing solution against test cases
- Identifying errors, bugs, or logical issues
- Debugging and correcting mistakes in implementation or logic
- Verifying correctness of the solution

**Example indicators:** "Let's trace...", "Testing with...", "This fails because...", "Walking through the example...", "The output should be..."

---

### 6. KNOWLEDGE RETRIEVAL (KR)
Retrieving and applying relevant knowledge.

**This category includes:**
- Recalling relevant algorithms, patterns, or similar problems
- Referencing library, API, or framework usage
- Recalling programming language syntax or features
- Applying domain knowledge or best practices

**Example indicators:** "I recall that...", "This is similar to...", "We can use the library...", "The syntax for...", "This pattern is..."

---

### 7. META-REASONING (MR)
Reflecting on and monitoring the reasoning process itself.

**This category includes:**
- Self-questioning or challenging own assumptions ("But wait...", "What if...")
- Validating correctness of current reasoning before proceeding
- Considering alternative approaches after initial solution
- Correcting identified mistakes in reasoning or logic
- Expressing confidence, uncertainty, or doubt
- General reflection on the reasoning process

**Example indicators:** "Wait...", "Actually...", "Let me reconsider...", "However...", "I'm not sure...", "Let me verify...", "On second thought..."

---

### 8. AUXILIARY (AUX)
Auxiliary statements that support the reasoning flow.

**This category includes:**
- Restating or summarizing previous steps
- Transitional statements between reasoning phases
- Side notes or observations not directly advancing the solution
- Organizational or meta-comments about the process

**Example indicators:** "To summarize...", "In other words...", "Note that...", "Moving on...", "As mentioned..."

---

## Segmentation Guidelines

### Core Principles
1. **Semantic Coherence**: Each step must represent a single, complete reasoning action within one category
2. **Natural Boundaries**: Split at category transitions, topic shifts, or completion of sub-tasks
3. **Completeness**: Include ALL content; no overlaps, no omissions

### Boundary Indicators

**Strong Boundaries** (almost always split here):
- **Category transitions**: Problem understanding → Solution design → Implementation planning
- **Explicit transitions**: "Now let's...", "Next, we...", "However,", "But wait,", "Moving to..."
- **Phase completion**: Finished understanding problem → Starting to design solution
- **Brief meta-reasoning (MR) followed by different activity**: 
  - "But wait, is this correct? Yes." (MR) → "Let me test..." (VV) → **Split**
  - "Actually, I need to reconsider..." (MR) → "The new approach is..." (SD) → **Split**

**Moderate Boundaries** (split if semantically distinct):
- Shift from exploring approaches to selecting one (within SD)
- Shift from planning to actual coding (IP → CC)
- Starting to test after implementation (CC → VV)
- New example being analyzed (within VV or PU)

**Weak Boundaries** (keep together in same step):
- Continuing the same trace or test case
- Elaborating on the same design point
- Short clarifications within the same reasoning action
- Self-questioning that continues within the same category without changing activity

### Granularity Guidance
- **Typical range**: 50-200 words per step (flexible based on content)
- **Do NOT** force split/merge to hit specific word counts
- **Prioritize** semantic integrity and category consistency
- A step should contain content from **only ONE category**
- **Short CoTs** may have 3-8 steps; **Long CoTs** may have 20-40 steps

### Special Cases

**Self-Questioning Patterns** ("But wait...", "However...", "Actually..."):
- **If questioning leads to category change** → **Split into two steps**
  - Example: "But wait, is this right? Yes." (MR) + "Let me test with N=1, M=7..." (VV) → **Split**
  - Reasoning: The brief self-check (MR) serves a different purpose than the detailed testing (VV)
  
- **If short self-reflection (< 2-3 sentences) precedes different activity** → **Split**
  - Example: "Wait, I made an error." (MR) + "The correct code is..." (CC) → **Split**
  
- **If questioning continues within same category without changing activity** → **Keep together**
  - Example: "But wait, we need to consider edge cases... [continues same design discussion]" (all SD) → **Keep as one SD step**
  - Example: "However, this approach has issues... Let me think of another way..." (all SD) → **Keep as one SD step**

**Example Analysis**:
- **Analyzing ONE example to understand problem** → One PU step
- **Tracing through ONE example to verify solution** → One VV step
  - Keep "Let me test with..." setup together with the detailed trace in one VV step
- **Analyzing MULTIPLE distinct examples** → Split into separate steps (one per example)
- **Brief verification statement + detailed test** → Split if they serve different purposes (MR + VV)

**Testing/Verification Sequences**:
- **Setup statement ("Let me test with...") + detailed verification** → Keep as **ONE VV step**
- **Brief self-check ("Is this right? Yes.") + new verification with examples** → **Split into MR + VV**
- **Multiple distinct test cases** → Split into separate VV steps (one per test case if semantically distinct)

**Code Sections**:
- **Discussing what code structure is needed** → IP
- **Writing actual code** → CC
- **Testing/verifying code with examples** → VV

**Transitional Phrases**:
- If transition phrase introduces a **new category or activity** → Split before the transition
- If transition phrase is just connecting thoughts **within same activity** → Keep together

---

## Classification Priority Rules

When a step could fit multiple categories, use this priority:

1. **MR** > others if the primary purpose is self-reflection, self-questioning, or monitoring own reasoning
2. **VV** > PU when tracing through an algorithm to verify it works (not just understanding the problem)
3. **IP** > SD when discussing specific implementation steps or details (not high-level strategy)
4. **SD** > IP when still at the design/exploration phase (not yet planning implementation details)
5. **PU** > SD when still clarifying what the problem asks (not yet thinking about solution)

---

## Output Format

Return a JSON array where each element represents one reasoning step:
```json
[
  {{
    "step_id": "1",
    "category": "PU",
    "content": "<exact text from the original CoT>"
  }},
  {{
    "step_id": "2",
    "category": "SD",
    "content": "<exact text from the original CoT>"
  }}
]
```

**Requirements**:
- `step_id`: Sequential numbers starting from "1"
- `category`: One of the 8 categories: PU, SD, IP, CC, VV, KR, MR, AUX
- `content`: Exact text from original CoT (preserve all text, no summarization, no modification)
- Output **ONLY** the JSON array, no explanations, no markdown code blocks, no preamble
